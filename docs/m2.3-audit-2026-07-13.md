# M2.3 audit summary

**State**: endpoint shipped and tested. Zero implementation gaps in the
signing path itself. Findings below are hygiene + URL correctness, not
functional breakage.

**Brief scope quote** (SHADOW_V3_BRIEF.md line 55):
> `POST /api/evidence/events` (batch), same contract as the library — for
> anything that can send JSON. Replaces/generalizes the old `/api/attest`.

## Findings (ranked by launch impact)

### 🔴 F1 — URL mismatch between docs and code

**Docs say**: `POST /api/evidence/events` (README.md:129, brief line 55)
**Code serves**: `POST /api/evidence-events` (file at `api/evidence-events.js`,
Vercel routes filename → path, dash preserved)

**Impact**: Anyone reading the README or brief and doing
`curl /api/evidence/events` hits 404. Not launch-blocker if the README
is the source of truth (fix by moving file), but launch-blocker if
someone integrated against the brief URL.

**Fix (recommended B)**:
- **A (5 min)**: Rewrite docs to `/api/evidence-events`. Loses brief-original
  slug, doesn't align with `packages/attest-core/`-style separation.
- **B (10 min)**: Move `api/evidence-events.js` → `api/evidence/events.js`.
  Vercel's file-based routing gives the slash automatically. Add
  `api/evidence-events.js` as a 308-redirect stub for back-compat if
  anyone integrated against the dashed form.

### 🟡 F2 — No max event count or body size guard

Current: unbounded. A 100k-event bundle passes structural validation and
gets signed, potentially exhausting memory + wasting Vercel invocation
seconds.

**Fix**: add `MAX_EVENTS_PER_REQUEST = 5000` and reject with 413 if exceeded.
The library-side 10k-event benchmark is a valid ceiling for real sessions;
5k is a defensive margin. Add matching test.

### 🟡 F3 — No idempotency key

Client retry on network flake creates a NEW `session_id` (if omitted) and
returns a NEW signed bundle with different `batch_root`. Downstream
storage collects duplicates that verify individually but diverge.

**Fix**: accept optional `Idempotency-Key` header (UUID). Server hashes it
into the session_id derivation so retries yield byte-identical bundles.
Standard REST pattern; matches Stripe, matches how banks deploy.

### 🟡 F4 — Response missing top-level `session_id`

Currently the client must parse `bundle.header.session_id` from the response
to correlate. Adding a top-level `session_id` key mirrors it out for common
tooling.

**Fix**: add one line to the 200 response.

### 🟢 F5 — Error format not yet in {seq, reason, impact} shape

Current: `{error: "...", reason: "..."}`. Verifier error format spec
(docs/spec/verifier-error-format.md) mandates `{seq, reason, impact}` for
verifier surfaces. The ingest endpoint is at the OPPOSITE end of the
pipeline (input validation, not verification), but consistency argues
for aligning.

**Fix**: defer to 2026-07-17 (Thursday) verifier error format port day.
Include this endpoint's error shape in the port. NOT a launch blocker.

### 🟢 F6 — No rate limiting at the endpoint

Vercel provides basic DDoS protection but there is no application-level
rate limit. A misconfigured client in a tight loop can rack up Vercel
invocation cost.

**Fix**: post-launch, either add Vercel Edge Config rate limit or document
"deploy behind a reverse proxy" in the endpoint doc. NOT a launch blocker.

### 🟢 F7 — Missing `X-Shadow-Bundle-Version` response header

Adding `X-Shadow-Bundle-Version: 2` to the response signals the wire format
version to older/newer clients without them having to parse the bundle.

**Fix**: one-line addition, nice-to-have.

## What's already good

- 11 tests passing, cover: 405 non-POST, 400 missing session, 400 unknown
  event_type, 400 unknown actor, 400 missing agent, 200 happy path with
  bundle-shape validation, environmental fingerprint required, chained
  events verify (round-trip).
- Env-based signing key with clear 500 remediation message.
- CORS headers correct for cross-origin submission.
- Cache-Control: no-store (correct — signed bundles are per-request).
- Structural validation happens BEFORE signing (fail cheap).
- Response includes `verify_hint` pointing users at the CLI + verify.html.
