# Determinism Floor

> A named procurement-defensibility pattern. The hardest-to-violate compliance thresholds are encoded as hardcoded conditionals with pinned tests, not as model output.

## The pattern

In regulated lending, certain thresholds are absolute. Below a credit-eligibility floor, no human override is appropriate. Above a fair-lending review trigger, no model judgment can skip the human reviewer. These are not "guidance for the AI"; they are non-negotiable invariants.

Treating these as **prompt instructions** ("please remember that FICO < 700 should be blocked") is unsafe. An LLM can be jailbroken, can drift, can hallucinate a threshold of 650 in a hot moment. The compliance officer who has to defend the decision in front of an examiner cannot point at a prompt.

Treating them as **hardcoded JS conditionals with pinned tests** makes them auditable in 8 lines of code. The examiner can read the source. The CI proves the threshold cannot regress. A future contributor cannot raise the FICO floor to 680 without the test failing.

This is what Shadow does.

## How Shadow implements it

### 1. FICO < 700 → hard block

`lib/run-loan-council.js` (Credit Fundamentals voice):

```js
if (loan.fico < CREDIT_THRESHOLDS.FICO_FLOOR) {
  return {
    verdict: "block",   // not "escalate" — Loredana's 2026-06-19 policy clarification
    rationale_short: `FICO ${loan.fico} below ${CREDIT_THRESHOLDS.FICO_FLOOR} eligibility floor`,
    adverse_action_codes: ["AA01"],  // CFPB Circular 2026-03 mapping
  };
}
```

The threshold is sourced from `lib/constants/credit-thresholds.js`:

```js
export const CREDIT_THRESHOLDS = Object.freeze({
  FICO_FLOOR: 700,
  DTI_MAX: 0.36,
  LTV_MAX: 0.80,
  VAR_MAX: 0.12,
});
```

Pinned by `test/loan-policy.test.js`:

```js
test("FICO 699 returns block, not escalate", () => {
  const result = runLoanCouncil({ fico: 699, dti: 0.20, ltv: 0.50, var: 0.05 });
  assert.equal(result.final_verdict, "block");
  assert.deepEqual(
    result.voices.find(v => v.voice_id === "credit").adverse_action_codes,
    ["AA01"]
  );
});
```

A contributor cannot raise the floor to 680 without the test failing. CI prevents the regression.

### 2. AA01–AA05 codes emit from a schema, with attribution

`lib/schemas/adverse-action.js` exports both the codes and their source:

```js
export const ADVERSE_ACTION_CODES = Object.freeze({
  AA01: "Credit score below underwriting floor",
  AA02: "DTI ratio exceeds policy ceiling",
  AA03: "LTV ratio exceeds policy ceiling",
  AA04: "VaR exceeds Risk Appetite Note ceiling",
  AA05: "Fair Lending Review triggered (CFPB / ECOA)",
});

export const AA_SOURCES = Object.freeze({
  AA01: "CFPB Circular 2026-03 §3.1 (credit-eligibility)",
  AA02: "CFPB Circular 2026-03 §3.2 (ability-to-repay)",
  // ...
});
```

The MCP tool `shadow_traceability` exposes this for read access at audit time.

### 3. Fair Lending Review flag → forced escalation

If `loan.fair_lending_review_flag === true`, the Fair Lending Compliance voice cannot return `approve` regardless of model output. The verdict resolver in `lib/run-loan-council.js` short-circuits:

```js
if (loan.fair_lending_review_flag && voices.fair_lending.verdict === "approve") {
  voices.fair_lending.verdict = "escalate";
  voices.fair_lending.adverse_action_codes.push("AA05");
}
```

This cannot be jailbroken because it runs **after** all 5 voices return, at the aggregator.

## What this gets you in procurement

- The examiner can read **8 lines of JavaScript** to verify FICO < 700 cannot be approved. Compare to reading 80 pages of Model Risk Management documentation around an ML score.
- The CI run on every PR proves the floor cannot be quietly raised or lowered. The threshold is "frozen in code" — a `Object.freeze()` call enforces it at runtime as well.
- AA01–AA05 codes are not generated by the LLM. They are emitted by the deterministic resolver based on which voice fired what. CFPB Circular 2026-03's specificity requirement is satisfied mechanically.

## What this does NOT replace

- A real bank's policy is more complex than 4 thresholds. Determinism Floor handles the **non-negotiable** thresholds. Soft thresholds (escalate paths, DTI nuances under compensating factors) still flow through the 5-voice deliberation.
- A bank will want to **change** the thresholds. Shadow exposes them via `LOAN_DEFAULTS` for per-deployment override; the deterministic enforcement pattern survives the threshold value change.

## Why competitors can't copy this on a roadmap

Zest AI auto-generates SR 26-2 (formerly SR 11-7) Model Risk paperwork **around** a probabilistic ML score. The floor of a Zest decision is the score's confidence interval, not a hard cutoff in code. To match Determinism Floor they would need to refactor away from "ML score wins" to "hardcoded rules can veto the ML score" — which is the exact pattern they sell against.

Anthropic FS templates have no thresholds in them at all. The model decides.

Hebbia is a research-search product; it doesn't make verdicts.
